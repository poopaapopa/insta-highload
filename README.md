## 3. **Глобальная балансировка**

#### 3.1 Домены

Instagram разделяет трафик на динамический (API) и статический (CDN) для оптимизации.

*   **Основной домен:** `i.instagram.com` (мобильный API), `www.instagram.com` (Web).
*   **CDN домены (Media):** `scontent.cdninstagram.com` (Фото/Видео). Используется Domain Sharding для обхода лимитов браузеров и гео-распределения.
*   **Graph API:** `graph.instagram.com` (Для сторонних разработчиков и бизнес-интеграций).

#### 3.2 Data Centers
Instagram — глобальный сервис с аудиторией 2+ млрд человек. В отличие от локальных сервисов, он требует размещения Origin-датацентров (где хранятся данные) на разных континентах.
Мы используем инфраструктуру Meta.

#### 3.3 Технологии глобальной балансировки

**DNS (Cartographer)**

Используем **Geo-DNS** ([**Cartographer** Meta](https://engineering.fb.com/2017/08/21/networking-traffic/steering-oceans-of-content-to-the-world/)) для направления пользователя на ближайший Edge Hub, а затем **Backbone Routing** для доставки запроса в Origin DC.

1.  Пользователь из Индии открывает приложение.
2.  Cartographer определяет, что пользователь находится в подсети определённого провайдера.
3.  Система знает, что сейчас маршрут до **Сингапура** перегружен, и направляет пользователя на Edge-ноду в **Мумбаи** (кэширующий прокси), которая по оптике соединена с Сингапуром.
4.  DNS возвращает IP ближайшей Edge-ноды.

При этом Edge отдает статику из локального кэша, а запрос API летит по внутренней оптике Meta в **Oregon DC (USA West)**.

**BGP Anycast**

Для обеспечения минимальной задержки и защиты от DDoS используем технологию **IP Anycast**.

Единый диапазон IP-адресов (например, для DNS-серверов и CDN-фронтов) анонсируется одновременно из всех PoP (точек присутствия) по протоколу BGP. → Глобальная сеть интернет автоматически направляет запрос пользователя к топологически ближайшему дата-центру (ользователь из Берлина → маршрутизируется в узел **Frankfurt Edge**.). В случае аварии в узле (например, Франкфурт недоступен), BGP-маршруты автоматически перестраиваются, и трафик из Берлина направляется в следующий ближайший узел (например, Амстердам) без необходимости изменения DNS-записей.

## 4. **Локальная балансировка**

Для локальной балансировки (по приходе в конкретный ДЦ) применим **двухуровневую архитектуру балансировки (L4 + L7)**, которая в связке с оркестратором **Kubernetes** обеспечит автомасштабирование и отказоустойчивость.

### 4.1 Layer 4 Load Balancing

На транспортном уровне при входе в ДЦ пакеты встречает L4-балансировщик. В масштабах Instagram классические прокси (вроде Nginx stream) не справляются с объемом пакетов, поэтому Meta использует программный балансировщик на базе eBPF/XDP (eXpress Data Path) - **Katran**.

Его ключевое преимущество заключается в том, что он работает на высокой скорости, перехватывая и перенаправляя сетевые пакеты на уровне драйвера сетевой карты, еще до того, как операционная система выделит под них память. Это позволяет Meta не использовать отдельные серверы под балансировку: Katran может работать фоном на любом сервере в ДЦ параллельно с другими задачами.

Katran использует технологию инкапсуляции IP-in-IP при пересылке пакетов внутри ДЦ для реализации подхода Direct Server Return (DSR). Балансировщик принимает входящий HTTP-запрос от клиента и пересылает его на нужный L7-прокси. Когда L7 формирует объёмные данные, этот ответ отправляется напрямую пользователю в обход балансировщика L4. Katran же позволяет этому случится, недавая IP адресу измениться при пересылке на L7 путём создания обёрточного пакета над существующим. Такая асимметричная маршрутизация полностью исключает Katran из пути исходящего трафика, гарантируя отсутствие перегрузок на входе.

*Источник*: [Open-sourcing Katran, a scalable network load balancer](https://engineering.fb.com/2018/05/22/open-source/open-sourcing-katran-a-scalable-network-load-balancer/).

### 4.2 Layer 7 Load Balancing

После прохождения транспортного L4 трафик попадает на уровень L7-прокси. Здесь также в отличие от многих компаний, использующих Nginx или Envoy, Meta использует собственную разработку под названием **Proxygen**. Это высокопроизводительный HTTP-фреймворк, написанный на C++. Изначально проект создавался именно как кастомный обратный прокси-сервер (отсюда и название), так как существующие на рынке решения не масштабировались под нужды компании и не обладали необходимой гибкостью для внедрения специфических функций, таких как проприетарные алгоритмы балансировки (с учётом загрузки ЦП и локализации данных) или ранняя поддержка WebSockets и TLS false start (ещё до появления QUIC).

Proxygen выделяется своей архитектурой: обработка разделена на сессии, кодеки, транзакции и обработчики. Это позволило фреймворку адаптировать новый протоколы HTTP/3 без необходимости дублирования или переписывания базового сетевого кода. Внутри Proxygen использует неблокирующий ввод-вывод и механизм Linux epoll, что делает сервер эффективнее по процессору и памяти. Также активное использование семантики перемещения из современного C++ позволяет передавать объёмные данные, характерные для медиаконтента Instagram, без лишнего копирования объектов в оперативной памяти и риска утечек.

Что же происходит на L7 уровне? Во-первых, это TLS Termination: Proxygen расшифровывает входящий безопасный трафик, снимая эту вычислительную нагрузку с серверов с бизнес-логикой (после чего общение внутри дата-центра идет уже по легкому внутреннему протоколу Thrift). Во-вторых, как уже говорилось выше, архитектурная гибкость Proxygen позволила Instagram одним из первых в мире внедрить QUIC, который снизил задержки при нестабильном интернете, избавив пользователей от блокировки начала очереди при потере пакетов. В-третьих, здесь происходит фильтрация: система интегрирована с внутренним мониторингом ODS, что позволяет нивелировать DDoS-атаки на границе сети.

*Источники*: [Introducing Proxygen, Facebook's C++ HTTP framework](https://engineering.fb.com/2014/11/05/production-engineering/introducing-proxygen-facebook-s-c-http-framework/), [How Facebook is bringing QUIC to billions](https://engineering.fb.com/2020/10/21/networking-traffic/how-facebook-is-bringing-quic-to-billions/).

### 4.3 Оркестрация

Вся бизнес-логика Instagram также, как и в большинстве компаний, упакована в контейнеры. Однако как и в других областях, вместо повсеместно использумого решения, Meta использует собственную систему управления кластерами под названием **Twine**. Он рассчитан на куда больший масштаб чем Kubernetes - Twine оркестрирует все дата-центры Meta по всему миру. Преимущество Twine заключается в распределении задач с учетом энергопотребления серверов, износа оборудования и суточных колебаний трафика.

Также система оркестрации тесно связана с балансировщиками L7 через механизм Health Checks. Twine постоянно мониторит состояние каждого контейнера. Если сервер физически выходит из строя или процесс зависает, оркестратор немедленно исключает его IP-адрес из таблиц маршрутизации Proxygen и перераспределяет входящие запросы на рабочие машины. Одновременно с этим Twine поднимает новые копии упавшего сервиса на свободных серверах. Это гарантирует отказоустойчивость сервиса как во время пиковых нагрузок, так и при выходе из строя немалого количества оборудования.

*Источник*: [Twine: A unified cluster management system](https://engineering.fb.com/2019/06/06/data-center-engineering/twine/).
